// CONTRIBUTING

Help build a better memory system. Here's how to get started.

[ SETUP ]
Get the codebase running locally.

Fork the repo and clone your fork:

git clone https://github.com/aivelikivodja-bot/elara-core.git cd elara-core python3 -m venv venv source venv/bin/activate pip install -e ".[dev]"

The [dev] install includes pytest, coverage tools, and other dev dependencies.

[ RUNNING TESTS ]
Verify everything works before making changes.

Run the full test suite:

pytest tests/ -v

All 91 tests should pass. Tests use isolated temp directories — no real data is touched.

Tip: Run pytest --cov=core --cov=daemon --cov=elara_mcp to see coverage reports. We aim for 80%+ coverage on new code.
[ PROJECT STRUCTURE ]
Where things live and what they do.
elara-core/ ├── core/ # Core library (paths, schemas, Elara class) ├── daemon/ # State engine (mood, presence, emotions, decay) ├── elara_mcp/ # MCP server + tool definitions │ └── tools/ # 34 tools across 11 modules ├── tests/ # Test suite (pytest) ├── docs/ # GitHub Pages site └── pyproject.toml # Package config

Key modules:

  • core/ — Data paths, JSON schemas, base Elara API
  • daemon/ — Mood engine, presence tracking, emotional state
  • elara_mcp/tools/ — Individual MCP tool implementations
  • tests/ — Unit and integration tests
Note: The MCP server entry point is elara_mcp/server.py. Tools are auto-discovered from the tools/ directory.
[ MAKING CHANGES ]
Workflow for contributing code.
  1. Fork the repo and create a branch from main
  2. Write tests for new functionality (test-driven is encouraged)
  3. Run the test suite before submitting — all tests must pass
  4. Keep commits focused — one logical change per commit
  5. Open a PR with a clear description of what changed and why
Commit messages: Use imperative mood ("Add feature" not "Added feature"). First line under 50 chars, body text wrapped at 72 if needed.

[ CODE STYLE ]
Conventions we follow.
  • Python 3.12+ — We use modern f-string syntax and type hints
  • Type hints where they add clarity (not required everywhere)
  • Docstrings on public functions (Google style preferred)
  • No unnecessary dependencies — Keep the install lightweight

We don't enforce strict linting, but aim for readability. If your code is clear and tested, it'll likely be accepted.

Formatting: We don't use Black or strict formatters. Just keep it clean and consistent with surrounding code.
[ REPORTING ISSUES ]
How to file bugs and feature requests.

Use the issue templates on GitHub:

  • Bug Report — Something broken? Tell us what happened.
  • Feature Request — Got an idea? Pitch it here.

Include your OS, Python version, and the full error message if applicable.

Before filing: Check the FAQ — many common issues are already solved there.
